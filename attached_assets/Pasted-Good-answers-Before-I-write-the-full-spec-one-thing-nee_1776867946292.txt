ARTIFACT 1: Product Brief — Broker Support Ticket System
Purpose
Give brokers on Lendry a single, low-friction way to get help, report bugs, or request features. Give the lender (Carlos, as super admin) a single inbox to triage, respond, and track issues. Tightly integrated with the AI help bot so the bot handles volume and tickets only capture what matters.
Actors

Broker (submitter): logged into the portal. Opens tickets, uploads attachments, sees their own ticket history, receives updates.
Super Admin / Lender (Carlos): reviews and responds to all tickets, triages bugs, manages feature request backlog.

Core Concepts
1. Ticket Types (3)

Help Question: broker needs an answer or explanation. Resolved by a written reply.
Bug Report: software defect. Resolved when fix is shipped or confirmed non-reproducible.
Feature Request: product suggestion. Acknowledged quickly, reviewed when Carlos has time, marked as Under Review / Planned / Declined.

2. Ticket Lifecycle
Shared states across all types:

Open — newly submitted, not yet responded to.
In Progress — super admin has engaged; work is happening.
Waiting on Broker — super admin replied and is waiting on more info/confirmation.
Resolved — super admin marked resolved; broker can reopen within 14 days by replying.
Closed — finalized (auto-close after 14 days in Resolved with no broker reply).

Feature requests have additional sub-states: Under Review, Planned, Declined, Shipped. (No Kanban board in v1; these are just status values in the list view.)
3. Intake Flow (broker side)
The intake is bot-first:

Broker clicks "Get Help" button (persistent in portal UI).
Chat window opens with the AI help bot (the one we already speced).
Bot attempts to answer within its scope.
At any point, broker can click "This didn't solve my problem — open a ticket" OR the bot itself offers the escalation button when it detects it can't help.
On escalation, broker picks ticket type (3 cards: Help / Bug / Request).
Type-specific form appears, pre-filled with the bot conversation transcript.
Broker fills required fields, attaches files, submits.
Confirmation screen shows ticket ID, expected response time, and link to view ticket.

Brokers can also bypass the bot entirely if they want ("Skip to ticket form" link), but the default is bot-first.
4. Type-Specific Fields
All tickets share:

Subject (auto-suggested from bot convo if escalated, editable)
Description (required)
Attachments (optional, multiple files; encouraged)
Status, created_at, updated_at
Broker info (auto-attached)
Bot conversation transcript (if escalated from bot)

Help Questions additionally:

Category dropdown (Deal submission / Documents / Pricing / Messaging / Account / Other)

Bug Reports additionally:

Severity (Blocker / Major / Minor / Cosmetic) — broker picks, lender can reclassify
Steps to reproduce (required)
Expected behavior
Actual behavior
Page/URL where it happened (auto-captured from where they clicked the button, editable)
Browser + OS (auto-detected, editable)
Screenshot (strongly nudged — "A screenshot helps us fix this 3x faster")

Feature Requests additionally:

Use case / problem being solved (required — this filters low-quality requests)
Priority to broker (Nice to have / Important / Critical to my workflow)

5. Response Time Commitments (shown to broker on submission)

Help Questions: within 4 business hours
Bug Reports: within 24 hours
Feature Requests: reviewed within 5 business days (acknowledgment, not resolution)

6. Notifications
Broker receives (email + in-app):

Ticket submitted (confirmation + ticket ID + expected response time)
Super admin replies
Status changes to Resolved
Status changes to Planned or Shipped (feature requests only)

Super Admin (Carlos) receives:

In-app: notification for every new ticket
Email: for every new ticket
Email + SMS: immediately on new bug reports, regardless of severity (per your instruction)
Daily digest email: open tickets summary, oldest unresolved, anything breaching SLA
Broker replies: in-app + email

7. Super Admin View
Unified ticket inbox in super admin dashboard with:

Filters: type, status, severity (bugs), date range, broker, unread-to-me.
Sort: newest first (default), oldest first, most recent activity, SLA urgency.
List columns: type icon, subject, broker name, status, severity (if bug), last updated, age.
Quick actions from list: change status, reply, archive.

Per-ticket detail view:

Full thread (broker + admin messages)
Attachments gallery
Bot conversation transcript (if escalated) — collapsible
Broker context sidebar: name, company, recent deals, last login, total ticket history
Internal notes field (admin-only, never shown to broker)
Reply box (public reply to broker)
Status + severity controls
Timeline of status changes

Dashboard widgets:

Count of open tickets by type
Count of bugs by severity
SLA breaches (tickets past their response window without a reply)
Oldest unresolved ticket
Trend graph: tickets submitted per week, last 8 weeks

8. Broker View of Their Own Tickets
"My Tickets" page in broker portal:

List of all their tickets with status, type, last updated
Filter by status
Click in to see full thread, reply, add attachments, view status history
Reopen a Resolved ticket by replying within 14-day window

9. AI Bot Integration (the escalation handoff)
When the bot can't help, it offers an "Open a support ticket" button. On click:

Full bot conversation transcript captures (timestamps, messages both sides)
Transcript attaches to new ticket invisibly (broker can see it if they expand; admin always sees it)
Bot's best guess at ticket type pre-selects (e.g. if broker said "this is broken," bot suggests Bug Report)
Description field pre-fills with a bot-generated summary: "Broker asked about X. I attempted Y. Issue unresolved because Z."
Broker reviews, edits if needed, submits

Bot's detection of "can't help" triggers:

User explicitly asks ("how do I contact support?" "this isn't working")
Bot's attempted answers followed by user frustration signals ("still not working," "that didn't help")
Bot detects question outside scope (specific borrower qualification, legal advice, etc.)
User clicks manual escalation button

10. File Attachments

Supported: images (PNG, JPG, GIF), PDFs, videos up to 50MB (helpful for bug reports with screen recordings), common doc types
Max 10 files per ticket, 50MB each
Stored in your existing file storage (likely same system as deal documents)
Virus scanned on upload (use existing infra if available, otherwise simple file type validation for v1)
Inline preview in admin view for images and PDFs

Compliance / Audit

All ticket activity logged with timestamps and actor ID
Broker cannot delete tickets (only archive from their view)
Admin can archive but not hard-delete (audit trail preserved)
Attachments retained per your standard data retention policy

Out of scope for v1 (explicitly deferred)

Kanban board for feature requests (list view only)
Public roadmap / customer-visible feature vote counts
Ticket assignment / queues (only one admin for now: Carlos)
Canned responses / macros
CSAT surveys after resolution
Integration with external dev trackers (GitHub, Linear, Jira) — tickets live only in Lendry
Analytics beyond the dashboard widgets listed
Multi-lender tenancy for tickets (Sphinx only for now)
Auto-translation for non-English submissions

Success metrics

Broker can submit a ticket in under 60 seconds (bot → escalate → form → submit).
AI bot resolves ≥50% of broker questions without ticket creation.
Average time-to-first-response under target SLAs.
Zero missed bug reports (every bug-type ticket acknowledged within 24h).
